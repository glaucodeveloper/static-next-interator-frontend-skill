Vanilla JavaScript · DOM nativo · contrato explicito

Seu DOM, com um contrato que continua legivel.

Uma skill para construir interfaces sem framework em que cada componente controla seu proprio root, seus eventos locais e suas atualizacoes — sem transformar o navegador em detalhe de implementacao.

O contrato central

Pequeno o bastante para auditar. Forte o bastante para orientar.

Um componente e um objeto vivo, nao uma string HTML nem um iterator ECMAScript completo. Ele expoe id, element e next(input); estado local e opcional.

  • Cria um novo root por atualizacao, usando <template>.
  • Conecta handlers antes de publicar o novo elemento.
  • Preserva ownership: eventos locais ficam no componente; mensagens globais seguem ao Interator.
  • Retorna { value: Element, done: false } em cada ciclo.
const counter = CounterComponent({
  id: "counter",
  props: { counting: 0 },
});

document.querySelector("#app")
  .append(counter.next().value);

counter.next({ counting: 1 });

Anatomia canônica

Um generator é a instância viva do componente.

Este é o ciclo React-like do módulo original: o generator fecha o estado, devolve montagem e eventos, renderiza HTML e recebe o patch da próxima atualização.

  1. create(id) registra o generator em frontends[id].
  2. O primeiro yield devolve mount.
  3. O segundo expõe events[id].
  4. O loop devolve o template HTML com o estado atual.
  5. O handler chama frontend.next(patch) e aplica a string por outerHTML.
const frontend = Component.frontend(id);
Component.frontends[id] = frontend;

const mount = frontend.next().value;
Component.events[id] = frontend.next().value;

mount(document.body);

// dentro do handler
element.outerHTML = frontend.next({
  counting: nextValue,
}).value;

// dentro de *frontend(id)
while (true) {
  const patch = yield `<div id="${id}">...</div>`;
  state = Object.assign(state, patch || {});
}
1. Namespace e registro da instância

window.CounterComponents conserva todos os generators por id. frontends[id] é a instância persistente; events[id] é a superfície pública chamada pelo template.

window.CounterComponents = window.CounterComponents || {};

const CounterComponent = {
  frontends: window.CounterComponents,
  events: {},
};
2. Generator, estado fechado e mount

O generator preserva state entre renders. O primeiro yield retorna a função que pede o primeiro template e o insere no alvo.

*frontend(id) {
  let state = { counting: 0 };

  yield function mount(target = document.body, position = "beforeend") {
    const frontend = CounterComponent.frontends[id];
    target.insertAdjacentHTML(position, frontend.next().value);
  };
3. Handlers: template → events[id] → next(patch)

O segundo yield expõe handlers. Eles localizam o root atual, avançam o generator com o patch e substituem somente esse root.

yield {
  addCountingState(number) {
    const frontend = CounterComponent.frontends[id];
    const element = document.querySelector(`#${CSS.escape(id)}`);
    if (!frontend || !element) return;
    element.outerHTML = frontend.next({ counting: number }).value;
  },
};
4. Template e auto-update do estado

O loop produz a string HTML. Quando o handler chama next(patch), o patch entra em newState; depois o generator atualiza state e no próximo avanço produz HTML novo.

while (true) {
  const newState = yield `<div id="${id}">
    <span>${state.counting}</span>
    <button onclick="CounterComponent.events[id].addCountingState(${state.counting + 1})">
      click!
    </button>
  </div>`;

  state = Object.assign(state, newState || {});
}
5. create(id): bootstrap completo

create avança os dois yields iniciais, armazena mount e handlers e devolve a montagem pronta para o consumidor.

create(id) {
  const frontend = this.frontend(id);
  this.frontends[id] = frontend;
  const mount = frontend.next().value;
  this.events[id] = frontend.next().value;
  return mount;
}

CounterComponent.create("counter-1")(document.body);

Por que existe

Sem handlers inline. Sem estado escondido. Sem render global acidental.

Frontends pequenos tendem a acumular interpolacao insegura, listeners que se perdem e mudancas diretas no documento. A skill oferece limites praticos para manter JavaScript puro compreensivel ao longo do tempo.

Generators de componentes

Cada instância mantém seu generator, templates HTML e handlers registrados por id.

Eventos com ownership

Acoes locais religam seus listeners a cada root; rota, sessao e sincronizacao se tornam mensagens explicitas.

Tipos em runtime

Validadores estritos formalizam patches, modelos e resultados de next() em JavaScript puro.

01 · checkout

Checkout com estado local

Campos, validação, resumo do pedido e confirmação ocupam o mesmo componente sem misturar eventos brutos no estado.

Selecione um cenário para trocar apenas este preview e seu código.

function* CheckoutTemplate({ id, props = {} }) {
  let state = { email: "", payment: null, ...props };
  while (true) {
    const template = document.createElement("template");
    template.innerHTML = "<section data-checkout></section>";
    const statePatch = yield template.content.firstElementChild;
    state = { ...state, ...statePatch };
  }
}
events[id] = {
  update(patch) {
    const frontend = frontends[id];
    const element = document.querySelector(`#${CSS.escape(id)}`);
    if (frontend && element) {
      element.outerHTML = frontend.next(patch).value;
    }
  },
};
ação local→statePatch→novo root + commit

02 · evento local

O DOM não entra no estado

O listener lê event.currentTarget, calcula um patch pequeno e chama uma função do próprio componente. O objeto Event nunca é mesclado em state.

handleAdd(event) {
  const amount = Number(event.currentTarget.dataset.amount);
  return this.next({
    counting: this.state.counting + amount,
  });
}

03 · mensagem global

O Interator decide o alcance

Rota, sessão e sincronização pertencem ao app. A delegação normaliza a intenção e só então solicita a atualização da região afetada.

const message = {
  type: target.dataset.message,
  route: target.dataset.route,
};

const result = interator.dispatch(message);
if (result.changed.includes("route")) {
  renderRoute(root, interator, children);
}

04 · driver gerador

Composição não toma o ciclo

A função frontend() apenas ordena steps. O driver cria as instâncias, monta a primeira renderização e mantém o estado local dentro do componente.

function runApp(frontend, executeStep) {
  const app = frontend();
  let cursor = app.next();

  while (!cursor.done) {
    cursor = app.next(executeStep(cursor.value));
  }
}

Contrato de runtime

Falhe cedo quando um patch não pertence ao componente.

A validação não substitui uma tipagem estática quando o projeto precisa dela. Ela protege as fronteiras que JavaScript puro deixa abertas: campos extras, números inválidos, ausência de dados obrigatórios e resultados fora do protocolo.

const allowedKeys = new Set(["counting", "label"]);

for (const key of Object.keys(statePatch)) {
  if (!allowedKeys.has(key)) {
    throw new TypeError(`state.${key} não existe`);
  }
}

if (!Number.isFinite(nextState.counting)) {
  throw new TypeError("counting deve ser finito");
}

Laboratório de cenários

Nove exemplos para situações que aparecem em aplicações reais.

Abra cada cenário para ver o contrato em contexto. Os exemplos são deliberadamente explícitos: deixam claro quem possui o estado, quem recebe a mensagem e onde o teardown acontece.

01 · Formulário controlado com validação incremental

O componente aceita somente patches conhecidos. A validação acontece antes do commit e o texto entra no DOM por propriedades, não por interpolação.

next(statePatch = {}) {
  const allowed = new Set(["email", "touched", "error"]);
  if (Object.keys(statePatch).some(key => !allowed.has(key))) {
    throw new TypeError("patch de formulário desconhecido");
  }

  const nextState = { ...this.state, ...statePatch };
  if (typeof nextState.email !== "string") {
    throw new TypeError("email deve ser string");
  }

  Object.assign(this.state, statePatch);
  const element = renderForm();
  element.querySelector("input").value = this.state.email;
  element.querySelector("[data-error]").textContent = this.state.error;
  return commitComponentElement(this, element);
}
02 · Lista filtrável com estado derivado

O estado guarda a consulta e os itens fonte. A lista filtrada é derivada durante a renderização; não há uma segunda fonte de verdade.

next({ query = this.state.query } = {}) {
  if (typeof query !== "string") throw new TypeError("query inválida");
  this.state.query = query;

  const visible = this.state.items.filter(item =>
    item.name.toLocaleLowerCase().includes(query.toLocaleLowerCase())
  );
  const element = renderListShell();
  const list = element.querySelector("[data-list]");
  for (const item of visible) {
    const row = document.createElement("li");
    row.textContent = item.name;
    list.append(row);
  }
  return commitComponentElement(this, element);
}
03 · Preservação explícita de foco após replaceWith()

Substituir o root remove o foco. Quando a experiência depende dele, capture a intenção antes do commit e restaure-a depois de publicar o novo root.

const activeSlot = this.element?.contains(document.activeElement)
  ? document.activeElement.dataset.slot
  : null;

const result = commitComponentElement(this, element);

if (activeSlot) {
  result.value.querySelector(`[data-slot="${activeSlot}"]`)?.focus();
}

return result;
04 · Busca assíncrona com cancelamento e dispose()

Requests não pertencem a connect(). O componente mantém o controller como recurso próprio e o encerra quando for descartado definitivamente.

async search(term) {
  this.request?.abort();
  this.request = new AbortController();
  this.next({ status: "loading", term });

  const response = await fetch(`/api/search?q=${encodeURIComponent(term)}`, {
    signal: this.request.signal,
  });
  const items = await response.json();
  return this.next({ status: "ready", items });
},

dispose() {
  this.request?.abort();
  this.request = null;
}
05 · Subscription externa com ownership e teardown

Listeners de window, document, timers e subscriptions ficam fora do ciclo de renderização. A instância possui uma operação de início e outra de descarte.

start() {
  if (this.unsubscribe) return;
  this.unsubscribe = sessionStore.subscribe(snapshot => {
    this.next({ user: snapshot.user });
  });
},

dispose() {
  this.unsubscribe?.();
  this.unsubscribe = null;
}
06 · Rota global e atualização apenas do outlet

O root do app delega a intenção. O Interator devolve quais átomos mudaram, e o driver atualiza a região correspondente sem remontar o app inteiro.

root.addEventListener("click", event => {
  const target = event.target.closest("[data-route]");
  if (!target || !root.contains(target)) return;

  const result = interator.dispatch({
    type: "navigate",
    route: target.dataset.route,
  });

  if (result.changed.includes("route")) {
    renderRoute(outlet, children, result.snapshot.route);
  }
});
07 · Mutação otimista com rollback explícito

O componente pode exibir uma intenção pendente, mas a confirmação remota precisa ter caminho de sucesso e falha claro; nenhum erro é silenciosamente absorvido.

async toggleFavorite(id) {
  const previous = this.state.favorites;
  const favorites = toggle(previous, id);
  this.next({ favorites, pendingId: id, error: "" });

  try {
    await api.saveFavorites(favorites);
    return this.next({ pendingId: null });
  } catch (error) {
    return this.next({
      favorites: previous,
      pendingId: null,
      error: "Não foi possível salvar. Tente novamente.",
    });
  }
}
08 · Componente read-only com input de renderização

Um componente sem estado local ainda é vivo porque preserva element. Ele recebe dados de renderização, publica seu root e compartilha o mesmo commit.

function StatusBadge({ id }) {
  return {
    id,
    element: null,
    next({ label = "", tone = "neutral" } = {}) {
      const template = document.createElement("template");
      template.innerHTML = "<span data-label></span>";
      const element = template.content.firstElementChild;
      element.id = this.id;
      element.dataset.tone = tone;
      element.querySelector("[data-label]").textContent = label;
      return commitComponentElement(this, element);
    },
  };
}
09 · Modelos estritos para dados recebidos

Contratos de runtime são úteis nas bordas: payloads de API, definições de rota e mensagens. Campos extras e números não finitos devem falhar antes de alcançarem a UI.

const Route = define("Route", {
  name: Type.string,
  path: Type.string,
  authenticated: Type.boolean,
});

const route = new Route({
  name: "favorites",
  path: "/favorites",
  authenticated: true,
});

route.extra = "não permitido"; // lança TypeError

Escolha consciente

Uma alternativa a um framework completo, nao uma imitacao dele.

Quando faz sentido usar?

Ferramentas internas, paginas interativas, prototipos duraveis e migracoes graduais de JavaScript legado que pedem DOM nativo e atualizacoes localizadas.

O que a skill nao promete?

Ela nao promete compatibilidade com o protocolo Iterator e nao esconde efeitos globais dentro de next(). Timers, subscriptions e listeners de window ou document exigem ownership e dispose() explicitos.

Comunidade

Traga casos reais. Fortaleca o contrato.
