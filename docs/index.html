O contrato central

Pequeno o bastante para auditar. Forte o bastante para orientar.

Um componente e um objeto vivo, nao uma string HTML nem um iterator ECMAScript completo. Ele expoe id, element e next(input); estado local e opcional.

  • Cria um novo root por atualizacao, usando <template>.
  • Conecta handlers antes de publicar o novo elemento.
  • Preserva ownership: eventos locais ficam no componente; mensagens globais seguem ao Interator.
  • Retorna { value: Element, done: false } em cada ciclo.
const counter = CounterComponent({
  id: "counter",
  props: { counting: 0 },
});

document.querySelector("#app")
  .append(counter.next().value);

counter.next({ counting: 1 });

Por que existe

Sem handlers inline. Sem estado escondido. Sem render global acidental.

Frontends pequenos tendem a acumular interpolacao insegura, listeners que se perdem e mudancas diretas no documento. A skill oferece limites praticos para manter JavaScript puro compreensivel ao longo do tempo.

Componentes nextable

Estado local, identidade de DOM e substituicao segura do root convivem no mesmo objeto.

Eventos com ownership

Acoes locais religam seus listeners a cada root; rota, sessao e sincronizacao se tornam mensagens explicitas.

Tipos em runtime

Validadores estritos formalizam patches, modelos e resultados de next() em JavaScript puro.

01 · componente stateful ao vivo

Um root por atualização

Os botões chamam next(statePatch). O componente cria um novo elemento, liga o listener e substitui o root anterior.

Aguardando a primeira renderização.

function Counter({ id, props = {} }) {
  return {
    id,
    state: { counting: props.counting ?? 0 },
    element: null,

    next(statePatch = {}) {
      Object.assign(this.state, statePatch);

      const template = document.createElement("template");
      template.innerHTML = "<output></output>";
      const element = template.content.firstElementChild;
      element.id = this.id;
      element.textContent = this.state.counting;
      element.component = this;

      if (this.element?.isConnected) {
        this.element.replaceWith(element);
      }
      this.element = element;
      return { value: element, done: false };
    },
  };
}

02 · evento local

O DOM não entra no estado

O listener lê event.currentTarget, calcula um patch pequeno e chama uma função do próprio componente. O objeto Event nunca é mesclado em state.

handleAdd(event) {
  const amount = Number(event.currentTarget.dataset.amount);
  return this.next({
    counting: this.state.counting + amount,
  });
}

03 · mensagem global

O Interator decide o alcance

Rota, sessão e sincronização pertencem ao app. A delegação normaliza a intenção e só então solicita a atualização da região afetada.

const message = {
  type: target.dataset.message,
  route: target.dataset.route,
};

const result = interator.dispatch(message);
if (result.changed.includes("route")) {
  renderRoute(root, interator, children);
}

04 · driver gerador

Composição não toma o ciclo

A função frontend() apenas ordena steps. O driver cria as instâncias, monta a primeira renderização e mantém o estado local dentro do componente.

function runApp(frontend, executeStep) {
  const app = frontend();
  let cursor = app.next();

  while (!cursor.done) {
    cursor = app.next(executeStep(cursor.value));
  }
}

Contrato de runtime

Falhe cedo quando um patch não pertence ao componente.

A validação não substitui uma tipagem estática quando o projeto precisa dela. Ela protege as fronteiras que JavaScript puro deixa abertas: campos extras, números inválidos, ausência de dados obrigatórios e resultados fora do protocolo.

const allowedKeys = new Set(["counting", "label"]);

for (const key of Object.keys(statePatch)) {
  if (!allowedKeys.has(key)) {
    throw new TypeError(`state.${key} não existe`);
  }
}

if (!Number.isFinite(nextState.counting)) {
  throw new TypeError("counting deve ser finito");
}

Escolha consciente

Uma alternativa a um framework completo, nao uma imitacao dele.

Quando faz sentido usar?

Ferramentas internas, paginas interativas, prototipos duraveis e migracoes graduais de JavaScript legado que pedem DOM nativo e atualizacoes localizadas.

O que a skill nao promete?

Ela nao promete compatibilidade com o protocolo Iterator e nao esconde efeitos globais dentro de next(). Timers, subscriptions e listeners de window ou document exigem ownership e dispose() explicitos.

Comunidade

Traga casos reais. Fortaleca o contrato.

Contribuicoes de exemplos, migracoes e validacoes sao bem-vindas quando preservam a ownership do componente e deixam o ciclo mais verificavel. Antes de enviar uma mudanca, execute node scripts/validate.mjs.
