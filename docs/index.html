Antes do primeiro exemplo

Uma interface inteira reduzida a uma sequência de uma função

Bibliotecas de interface frequentemente distribuem uma única atualização entre estado, hooks, scheduler, reconciliação, bindings e convenções de build. Static Next escolhe outra direção: conservar a complexidade do produto, mas tornar explícito o mecanismo que a executa.

O componente não descreve apenas como a tela parece. Ele conserva sua instância em this, expõe métodos ao próprio root, produz o elemento atual, suspende e espera a informação mínima necessária para continuar. Essa informação é o StatePatch.

A

Estado é dado visível

this.state contém aquilo que determina a renderização. Não há estado duplicado em registries ou closures externos ao componente vivo.

B

O root é a superfície da instância

element.component = this permite que o DOM encontre os métodos da instância atual sem um barramento global de callbacks.

C

Yield é a fronteira

Na ida, entrega o HTMLElement. Na retomada, recebe o patch passado a next(). Uma expressão, duas direções.

D

Patch é a próxima intenção

O método não ordena um lifecycle oculto. Ele calcula uma alteração pequena e retoma o programa exatamente onde ele estava suspenso.

Como acompanhar o contador a seguir

Leia primeiro a criação do contexto; depois, a declaração de estado e métodos. Dentro do while, acompanhe a expressão de dentro para fora até o yield. Quando imaginar um clique, faça o caminho inverso: o método chama next(StatePatch), o assign pendente termina e a volta seguinte produz outro root.

contexto vivo→estado + métodos→root inline→yield→state patch→novo root

O contador é pequeno de propósito, mas não é um pseudocódigo incompleto: ele contém a anatomia que os sete exemplos posteriores preservam ao adicionar validação, coleções, foco, composição e assincronicidade.

Posicionamento no ecossistema

Static Next e Web Components resolvem problemas diferentes

Web Components oferecem um protocolo nativo para publicar novos elementos HTML, definir atributos observáveis, reagir à conexão com o documento e, quando necessário, encapsular marcação e estilos com Shadow DOM. Static Next não cria uma nova tag: ele organiza o estado local e a atualização de um root DOM como um programa generator explícito.

A diferença principal está na superfície de programação. Em um custom element, comportamento e atualização costumam atravessar classe, propriedades, atributos, callbacks de lifecycle e um método de render. Em Static Next, a linha temporal permanece concentrada em uma única function*: estado atual, root produzido, yield, next(StatePatch) e próximo root.

A

Identidade pública

Web Components usam uma tag registrada por customElements.define(). Static Next usa uma instância comum ligada ao root por element.component, sem registro global de nomes de elementos.

B

Lifecycle

Custom elements possuem callbacks como connectedCallback e attributeChangedCallback. Static Next usa o protocolo nativo do iterator: yield suspende e next(patch) retoma o mesmo programa.

C

Encapsulamento

Shadow DOM é uma vantagem clara quando o componente precisa isolar estilos, slots e detalhes internos. Static Next privilegia DOM aberto e rastreável; encapsulamento pode ser acrescentado quando o produto realmente exige essa fronteira.

D

Distribuição

Web Components são fortes como contrato HTML reutilizável entre aplicações e frameworks. Static Next é forte como modelo de estado local legível dentro de uma aplicação e também pode ser hospedado por um custom element.

Web Component→tag + lifecycle→atributos e propriedades→render interno→DOM encapsulado ou aberto
Static Next→estado local→yield HTMLElement→next(StatePatch)→novo root

A vantagem central de Static Next é a transparência temporal. Ele reduz a distância entre evento, alteração de estado e DOM seguinte. Web Components continuam sendo a escolha natural quando a prioridade é oferecer uma API HTML pública, interoperável e opcionalmente encapsulada.

Natividade primeiro · bundler depois

O navegador já contém o runtime principal

Static Next é composto por primitivas da própria plataforma: function*, iterators, Object.assign, <template>, DocumentFragment, HTMLElement, eventos DOM e replaceWith(). O componente não depende de um runtime de framework para existir, preservar estado ou atualizar sua região.

Isso permite executar a página diretamente no navegador, inspecionar o código sem uma etapa obrigatória de transformação e adotar build tooling somente quando ele entrega valor operacional. A natividade define o contrato; o bundler, quando presente, organiza a entrega.

1

Sem build obrigatório

Um arquivo HTML com JavaScript padrão já consegue iniciar o iterator, renderizar o root e receber patches. O ciclo do componente não depende de compilador, plugin ou transformação proprietária.

2

Sem runtime paralelo

O navegador executa o generator e manipula o DOM diretamente. Não existe scheduler, virtual DOM ou reconciliador intermediando a causalidade básica do componente.

3

Bundler como ferramenta

Vite, Rollup, esbuild, webpack ou outra ferramenta podem agrupar módulos, otimizar assets, minificar e preparar diferentes alvos. Nenhuma delas precisa redefinir o protocolo yield HTMLElement ↔ next(StatePatch).

4

Saída descartável

Como o código-fonte permanece JavaScript e DOM reconhecíveis, trocar ou remover o bundler não exige reescrever o modelo mental do componente. A ferramenta de build não se torna a arquitetura da interface.

JavaScript padrão→generator nativo→template + fragment→HTMLElement vivo→DOM do navegador
A

Execução direta

O mesmo contrato funciona em um script simples ou em módulos carregados pelo navegador.

<script type="module">
  const counter = component(Counter, {
    id: "counter-1",
  });

  document.querySelector("#app")
    .append(counter.next().value);
</script>
B

Build opcional

Adicione um bundler para necessidades de entrega, não para tornar o componente compreensível ou executável.

código-fonte nativo
  → módulos e assets
  → otimização opcional
  → navegador
  → o mesmo iterator vivo

O carro-chefe é a natividade. Um bundler pode melhorar distribuição, cache e tamanho final, mas permanece substituível. O programa de UI continua expresso em recursos padronizados da plataforma e pode ser lido sem reconstruir um runtime escondido.

Tutorial principal

Anatomia completa, parte por parte

O contador é o exemplo máximo mínimo: contém tudo que um componente stateful precisa. Interfaces maiores apenas ampliam o shape do estado, os métodos e o template.

handler→this.next(patch)→state→fragment→replaceWith→yield root
1

Binder mínimo

Cria um contexto independente, chama o generator com esse contexto como this e liga os métodos nativos do iterator.

2

Estado e métodos em this

id, state, element e handlers pertencem à instância, não a registries globais.

3

Template inline vira fragmento

O template descartável nasce dentro de Object.assign(document.createElement("template"), { innerHTML }). Seu .content é consumido imediatamente, sem variável local.

4

Root estendido

element.component = this liga o nó atual ao contexto vivo que contém métodos e estado.

5

Commit localizado

Se o root anterior está conectado, replaceWith() o troca. A referência this.element passa a apontar para o novo root.

6

Yield bidirecional

Entrega o elemento atual e recebe o patch enviado pela próxima chamada a next(patch).


function* CounterComponent({ id, props = {} }) {
  const initialState = {
    counting: props.counting ?? 0,
    label: props.label ?? "Contador",
  };

  this.id = id;
  this.state = { ...initialState };
  this.element = null;

  this.increment = amount => this.next({
    counting: this.state.counting + amount,
  });
  this.reset = () => this.next(initialState);

  while (true) {
    Object.assign(
      this.state,
      yield (this.element = ((element) => {
        element.id = this.id;
        element.component = this;

        if (this.element?.isConnected) {
          this.element.replaceWith(element);
        }

        return element;
      })(Object.assign(document.createElement("template"), {
        innerHTML: /* html */ `<section>
          <output>${this.state.counting}</output>
          <button onclick="document.getElementById('${id}').component.increment(1)">+1</button>
          <button onclick="document.getElementById('${id}').component.reset()">reset</button>
        </section>`,
      }).content.firstElementChild)),
    );
  }
}

Generator nativo · state patch · DOM vivo

O componente é um programa suspenso.

Uma única function* guarda estado em this, materializa um DocumentFragment descartável, entrega um HTMLElement vivo e recebe a próxima transição por next(StatePatch).

A estrutura foi desenhada para ser lida do mesmo modo por quem programa, por quem revisa e pela IA que colabora com ambos: cada causa e cada efeito permanecem visíveis na linha do componente.

function*yield HTMLElementnext(StatePatch)element.componentreplaceWith

Da aplicação ao último root

Generators definem programas; iterators mantêm sua execução viva

Uma function* não serve apenas para percorrer coleções. Ela pode representar um programa que conserva dados e posição entre eventos. O generator define a sequência; o iterator é seu controle vivo. yield output torna o momento atual observável, enquanto next(input) retoma exatamente o mesmo programa com a próxima informação.

1

Programa de aplicação

Rota, sessão ou workflow podem durar por muitos eventos. O iterator preserva o estado e o ponto de execução sem reconstruir o programa a cada transição.

2

Superfície de controle nativa

next continua, throw devolve uma falha ao ponto suspenso e return encerra. Não é necessário inventar outro protocolo para controlar a sequência.

3

Static Next é o leaf

O componente especializa a mesma ideia no nó terminal de UI: produz um HTMLElement, recebe um StatePatch local e substitui somente o próprio root.

function* stateProgram(initialState) {
  const state = { ...initialState };

  while (true) {
    Object.assign(
      state,
      yield Object.freeze({ ...state }),
    );
  }
}

const application = stateProgram({
  route: "home",
  authenticated: false,
});

const initial = application.next().value;
const authenticated = application.next({
  authenticated: true,
}).value;
const details = application.next({
  route: "details",
}).value;
programa da aplicação→rota · sessão · workflow→leaf component→um root DOM vivo

Leaf não significa visualmente pequeno. Um formulário, uma tabela ou um painel complexo continua sendo leaf quando constitui uma fronteira local: possui um estado, um root e um ciclo de substituição. Coordenação global pertence ao programa da aplicação; o yield do componente não vira barramento de mensagens.

O ponto decisivo

Object.assign(state, yield element)

A expressão completa une a saída da renderização atual à aplicação automática do próximo state patch.

A

Saída

component.next() avança até o yield e recebe { value: element, done: false }.

const first = counter.next();
app.append(first.value);
B

Entrada

component.next(patch) retoma o ponto suspenso. O argumento vira o valor de yield element.

counter.next({ counting: 10 });
// patch === { counting: 10 }

O primeiro next() suspende no yield e entrega o root. O próximo next(patch) retoma a expressão; seu argumento torna-se o segundo argumento de Object.assign, atualiza this.state e libera a próxima volta do render.

Filosofia de escrita

O código é a linha temporal do componente

O template inline não é um truque de concisão. Sua posição dentro do yield preserva a sequência exata: produzir a interface atual, suspender, receber o StatePatch, concluir a atualização e produzir a interface seguinte.

abre assign→cria root inline→yield suspende→next(patch)→assign conclui→novo render
1

Atualização pendente

Object.assign(this.state, ...) abre a operação, mas ainda não possui o segundo argumento.

2

Produção transitória

O template é criado, preenchido e consumido inline. Ele não recebe nome porque não possui identidade nem duração no componente.

3

Fronteira de suspensão

O root é entregue por yield. A chamada externa recebe o HTMLElement; o assign continua incompleto.

4

Recebimento do StatePatch

this.next(statePatch) retoma exatamente o ponto suspenso. O patch vira o valor da expressão yield (...).

5

Conclusão e repetição

O Object.assign finalmente atualiza o estado. O while continua e o próximo root nasce do estado novo.

Object.assign(
  this.state,
  yield (
    this.element = ((element) => {
      element.id = this.id;
      element.component = this;

      if (this.element?.isConnected) {
        this.element.replaceWith(element);
      }

      return element;
    })(
      Object.assign(
        document.createElement("template"),
        { innerHTML: html },
      ).content.firstElementChild
    )
  ),
);

Separar o template em preparação anterior ao auto-state quebra essa representação linear e sugere um lifecycle que não existe. Persistem somente o iterator, this.state e this.element; o template é descartável.

Tooling opcional

HTML template string extensions continuam compatíveis

A transparência do paradigma não impede uma boa experiência de edição. Extensões com language injection podem reconhecer o HTML dentro da template string e oferecer highlight, autocomplete, Emmet e formatação.

A

Anotação sem runtime

Prefira /* html */ antes do template literal. O comentário orienta tooling compatível e desaparece para a execução.

innerHTML: /* html */ `
  <section>${content}</section>
`
B

Tagged template opcional

html`...` também pode ser usado quando a tag é conhecida e documentada. Ela deve continuar produzindo o valor esperado por innerHTML.

innerHTML: html`
  <section>${content}</section>
`

Uma tag html não deve esconder escaping, sanitização, registro de instâncias ou outro lifecycle. Syntax highlighting é tooling; segurança e semântica continuam responsabilidades explícitas do componente.

Genealogia

Inspiração clara em Effect/Effect-TS

A influência está na separação entre descrever um programa suspenso e interpretá-lo explicitamente. Static Next especializa essa ideia para o DOM: a função generator descreve o ciclo, o binder dirige next/throw/return, o valor produzido é um HTMLElement e o valor de retomada é um StatePatch.

É inspiração conceitual, não dependência ou compatibilidade de API. O runtime usa somente JavaScript e DOM nativos.

Estratégia para produção com IA

Transparência tecnológica é uma escolha operacional

Quando o código é produzido ou modificado com IA, simplicidade deixa de ser apenas preferência estética. Ela reduz a distância entre o que foi solicitado, o que foi gerado, o que será executado e o que uma pessoa precisa auditar antes de publicar.

O segredo de leitura da IA e o seu é o mesmo: seguir a linha do programa — estado atual, root produzido, suspensão, StatePatch recebido, estado atualizado e próximo root.

1

Menos overhead cognitivo

A transição não precisa ser reconstruída a partir de hooks, scheduler, transforms, registries e convenções implícitas distribuídas.

2

Menos overhead de contexto

A IA precisa de menos conhecimento sobre versões, plugins e internals antes de compreender uma alteração local. O protocolo relevante está no componente.

3

Menos overhead arquitetural

Estado e intenção não atravessam adapters que apenas renomeiam ou transportam o mesmo patch entre camadas.

4

Uma auditoria compartilhada

Pessoa e IA podem fazer as mesmas perguntas no mesmo trecho: qual estado entrou, qual método foi chamado, qual patch saiu e qual root foi substituído?

Simplicidade não elimina rigor

Validação de patch, escaping, acessibilidade, foco, teardown, concorrência e testes continuam obrigatórios. A vantagem é que esses controles também permanecem explícitos. Ganhos de bundle ou runtime só devem ser declarados depois de medição; o ganho imediato é a redução do overhead de interpretação.
